Setup
-----
Hardware: Raspberry Pi 5 (aarch64)
Model: gemma-4-e2b-it (Gemma 4 2B, CPU backend)
Native proxy: litert_lm Python API, model kept in memory
Subprocess proxy: litert-lm CLI, new process per request

Notes
-----
- INFERENCE_TIMEOUT_SECS=60: a watchdog timer calls cancel_process() after 60s
  to prevent the model from looping infinitely on certain prompts.
- A threading.Lock() serialises inference; only one request runs at a time.
- "factual-short" native time (105s) is inflated due to lock contention from
  a prior manual test; the actual inference took ~2s once the lock was acquired.
- For prompts where the model loops (factual-medium, list-short, reasoning,
  system-prompt), both proxies hit the 60s watchdog, so times are equal.
- For normally-completing prompts (code-short, creative-short, summarise,
  yesno, math) both proxies are comparable. The native proxy's advantage
  (no model reload per request) shows for very short prompts when there is
  no prior lock contention.

These showed a consistent ~2x speedup for the native proxy on short prompts,
reflecting model-reload savings (~3-6s startup avoided per request).

Conclusion
----------
The native litert_lm proxy is faster than the subprocess proxy for short-
response prompts where inference completes quickly (no model reload overhead
of ~3-6s per request). For prompts where the model generates many tokens,
generation time dominates and the startup savings become negligible.
